## Typography

This phase inherits the project's existing type scale (Geist Variable) and adds prose typography via `@tailwindcss/typography`.

| Role | Size | Weight | Line Height | Where Used |
|------|------|--------|-------------|------------|
| Body / prose | 16px | 400 (regular) | 1.75 (prose default) | ReactMarkdown article content via `prose` class |
| Label / sidebar link | 14px | 400 (regular) | 1.5 | Sidebar anchor links |
| Heading / page h1–h2 | 20px+ (prose managed) | 600 (semibold) | 1.2 | Rendered markdown headings via `prose` class |
| App heading | 20px (`text-xl`) | 600 (`font-semibold`) | 1.2 | AppHeader "Feature Cost Calculator" title — unchanged |

Notes:
- Markdown prose typography is fully managed by `@tailwindcss/typography` `prose prose-neutral max-w-none` on the article wrapper. Do not override individual heading sizes — the prose plugin handles the full heading hierarchy.
- Sidebar h3-level links (`level: 3`) use `pl-4` (16px) indent to visually nest under their parent h2.
- Font: Geist Variable, already loaded via `@fontsource-variable/geist` in `index.css`. No new font imports needed.

## Sidebar Anchor Navigation Contract

Sidebar links use `#/docs/{id}` format (not bare `#doc-{id}`). This maintains the hash routing namespace discriminator (`/` prefix = route hash; no `/` prefix = base64url calculator state). Source: RESEARCH.md Pattern 5, STATE.md [v1.2 Routing].

Click handler pattern:
```typescript
onClick={(e) => {
  e.preventDefault();
  window.location.hash = `/docs/${entry.id}`;
  document.getElementById(entry.id)?.scrollIntoView({ behavior: 'smooth', block: 'start' });
}}
```

Deep-link scroll on mount: `DocsPage` must include a `useEffect` that reads `window.location.hash` on mount, extracts the section ID, and calls `scrollIntoView` if a valid `doc-` prefixed section ID is found. This handles ROUTE-03 deep-link behavior. Source: RESEARCH.md Open Question 3.

## Heading ID Contract

- rehype-slug with `prefix: 'doc-'` generates `id="doc-{github-slugger-slug}"` on each heading
- Sidebar `extractHeadings()` function produces the same IDs using a custom `toSlug()` that matches github-slugger's algorithm for the research doc's ASCII headings
- Only h2 and h3 headings are extracted for the sidebar (h1 is the document title; h4+ are too granular for navigation)
- Verify ID accuracy after install: log `extractHeadings(researchMd)` and compare to DevTools-inspected `id` attributes on rendered headings

## Module-Level Constants (Performance Contract)

All `remarkPlugins`, `rehypePlugins`, and `components` props for `<ReactMarkdown>` MUST be defined at module level — not inline in JSX. Inline definitions create new array/object references on every render, causing react-markdown to re-parse the full 849-line document on every parent re-render.

```typescript
// Correct — module level
const REMARK_PLUGINS = [remarkGfm] as const;
const REHYPE_PLUGINS = [[rehypeSlug, { prefix: 'doc-' }]] as const;
const COMPONENTS = { h1: ..., h2: ..., h3: ..., h4: ... };
```

Source: RESEARCH.md Pattern 2, Pitfall 1.
